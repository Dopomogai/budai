# 19 — Glossary

Canonical terms used throughout budai's documentation and tooling. Disagreements between docs are resolved by this file.

## Core entities

**Agent.** A running instance of a role. Has a runner, a model, an opaque ID, a worktree (for Implementers), and a transcript. Ephemeral.

**Role.** A markdown file declaring an agent type's mission, permissions, skills, and default model. Persistent. budai ships five: Planner, Implementer, Verifier, Judge, Librarian.

**Skill.** A named, versioned, reusable procedure that one or more roles can invoke. Examples: `build-task-bundle`, `peer-review`, `audit-docs`. Persistent.

**Workflow.** A markdown file orchestrating roles for a class of work. Examples: `ship-feature`, `fix-bug`, `refactor`, `audit-repo`. Persistent.

**Task.** A markdown file describing a unit of work. Has frontmatter (status, fan-out, dependencies) and body (objective, user story, acceptance criteria, eventually plan). Lives in `tasks/open/` while active, `tasks/archive/` once complete or abandoned.

**Run.** A single agent execution. Identified by `<run-id>`. Has a transcript, optional diff, optional evidence. Lives in `runs/<run-id>/`. Streamed to backend.

**Attempt.** A submitted result from an Implementer in a fan-out. Identified by an opaque ID (`attempt-A`, `attempt-B`, ...) within a council. Anonymized at write-time; de-anonymized via `mapping.json`.

**Council.** Per-task folder containing all attempts, all reviews, the de-anonymization mapping, and the verdict. Survives the task as durable record.

**Verdict.** The Judge's output: which attempt won, why, what others got wrong, outstanding concerns, auto-spawned follow-ups, attribution. Lives in `council/<task-id>/verdict.md`.

**Bundle.** A single self-contained markdown file the Implementer reads to know everything about a task. Has YAML frontmatter declaring contents + content body. Goal: 95% context coverage in one read. Token-budgeted.

**Plan.** The Planner's elaborate output appended to a task body. Includes approach, decomposition, file-by-file changes, risks, AC mapping, recommended fan-out. The contract the Implementer executes against.

## Coordination

**Fan-out.** When more than one Implementer attempts the same task in parallel. Each in an isolated worktree. Declared in task frontmatter (`fan-out: N`). Default 1.

**Worktree.** Per-Implementer isolated git worktree, created at `runs/<run-id>/worktree/`. The Implementer's sandbox. Cannot read other implementers' worktrees.

**Opaque ID.** An anonymizing identifier (e.g., `attempt-A`) used during peer review so the Judge cannot bias on authorship. The mapping to real instance metadata lives in `council/<task-id>/mapping.json`.

**Human gate.** A pause point where a human approves, revises, or rejects. Two by default: end-of-Planner (architecture review) and end-of-Judge (final result review). Auto-approve rules can bypass for trivial tasks.

**Message board.** Channels of async coordination posts under `.agents/messages/channels/`. Channels: `tasks`, `review`, `ops`, `decisions`. Threaded replies under `messages/threads/`.

## Memory

**Lesson.** A durable observation extracted from runs that should change future behavior. Promoted from per-run to per-role to repo-wide as it recurs.

**ADR (Architectural Decision Record).** A short markdown file in `memory/decisions/` documenting why a choice was made. ADRs are durable and rarely revised; they're reference, not prose.

**Untouchable.** A constraint listed in `local/untouchables.md` describing things that look weird but must not change without discussion. Read by every agent before making non-trivial changes.

**Glossary entry.** A repo-specific term defined in `local/glossary.md`. Loaded into every bundle for terms appearing in the task body.

## Discovery

**Index.** Auto-generated navigation aid in `.agents/index/`. Two tiers: `tree.md` (file paths only, cheap) and `detailed.md` (paths + file headers, used by the Briefer for relevance scoring).

**File header.** A six-field comment block at the top of every source file: `@purpose`, `@why`, `@role`, `@exports`, `@uses`, `@stability`, optional `@gotchas`. Source of truth for the detailed index.

**README bundle.** `docs/READMEs.md` — concatenated bundle of every README in the repo. Auto-generated by the Librarian sweep. Used by the Planner during context-gathering for cross-cutting tasks.

## Distribution

**Manifest.** Per-repo `manifest.yaml` declaring which budai version + which skills/roles/workflows to pull from the registry. Pin versions; declare overrides; declare local-only extensions.

**Registry.** Central git repo (`dopomogai/budai`) holding canonical `base/` content. Source of truth for what skills/roles/workflows exist and what their versions are.

**Base.** Read-only mirror of the registry's content, materialized into `.agents/base/` by `librarian sync`. Reproducible from manifest + registry.

**Local.** Repo-specific overlay at `.agents/local/`. Same-name files override base; new-name files extend base.

**Runner.** Thin shim translating budai role definitions into invocations of a specific agent platform (Claude Code, Codex, direct-Anthropic, direct-OpenAI). Lives in `runners/<name>.md`.

## Operations

**Preflight.** A script (`bin/preflight`) that validates repo state before agent work begins. Checks: no `.orig` files, no untracked secrets, all source files have headers, docs reference only present symbols, etc. Required before any role runs.

**Postflight.** A script (`bin/postflight`) run after a role declares done. Checks: tests pass, no leftover state, no broken docs, no untracked outputs that should be committed.

**Sweep.** The Librarian's per-task or scheduled maintenance run. Audits docs, regenerates index, promotes lessons, updates stats, posts daily delta.

**Promotion.** Moving a lesson, skill, or convention up a layer (run → role → repo → registry). Always involves explicit decision (auto-promotion is allowed only for high-confidence cases like file renames).

**Compute tier.** The model class an agent runs at: Haiku (light, mechanical), Sonnet (default), Opus (heavy reasoning). Set per-role in role frontmatter; can be overridden per-skill via `tier-override:`.

## Status fields

**Task status:** `open` | `planning` | `implementing` | `reviewing` | `done` | `abandoned` | `coordinator`. Lives in task frontmatter.

**Attempt status:** `submitted` | `passed` | `failed` | `winner`. Set by Verifier and Judge.

**Skill stability:** `stable` | `experimental` | `deprecated`. Set in skill frontmatter. Affects whether the Librarian recommends it for new tasks.

**File stability:** `stable` | `experimental` | `deprecated`. Set in file header. Affects how aggressively the Briefer pulls the file into bundles.

## What budai is NOT

**Not** an IDE plugin. Not a code-completion tool. Not a chat interface. Not a CI/CD pipeline. Not a deployment system. Not a project-management replacement (it works alongside whatever the team uses for non-code tasks).

budai is the operating system. The agents are the OS-equivalent of running processes. The codebase is the filesystem they read and write. Everything else is optional.
