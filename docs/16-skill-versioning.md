# 16 — Skill versioning

Skills, roles, and workflows are versioned with semver. This document expands the basics from `04-skills.md` and `05-workflows.md` into a complete reference: what counts as a breaking change, manifest pin syntax, the deprecation flow, and how stats discriminate by version.

The same versioning rules apply to skills, roles, and workflows. Throughout this doc, "skill" stands in for all three unless noted.

## Semver recap

```
MAJOR.MINOR.PATCH
```

- **PATCH** (1.4.2 → 1.4.3) — clarifications, edge-case handling, prompt tweaks. No I/O contract change.
- **MINOR** (1.4.x → 1.5.0) — new optional input/output, new owner role. Backward compatible.
- **MAJOR** (1.x → 2.0.0) — input/output contract change, removed inputs, semantic shift. Breaking.

When a version goes major, set `breaking-changes-from:` in the new version's frontmatter pointing at the last pre-break version. The Librarian uses this to flag manifest inconsistencies.

## What counts as breaking

A change is **major (breaking)** if any of these apply:

- An existing input is renamed, removed, or has its type changed.
- An existing output is renamed or removed.
- An existing output's structure changes (a JSON schema field disappears or changes type).
- The skill's preconditions become stricter (e.g., now requires the bundle to have a `priority:` field that wasn't required before).
- The skill's postconditions become weaker (e.g., the skill used to guarantee idempotency and no longer does).
- Error semantics change in a way callers might miss (e.g., a previously-thrown exception becomes a silent skip).
- The skill's owners list narrows (e.g., a skill previously available to Implementers is now Librarian-only).

A change is **minor (compatible)** if:

- A new optional input is added (existing callers ignoring it still work).
- A new output field is added (existing consumers ignoring it still work).
- A new owner role is added.
- Internal procedure changes that don't alter I/O.

A change is **patch** if:

- Prompt phrasing improves clarity but doesn't change behavior.
- Edge cases get handled that previously errored or silently misbehaved.
- Documentation in the body improves.
- Internal step numbering changes but the procedure is equivalent.

When in doubt, bump higher. Breaking a consumer with a "patch" labeled as such is much worse than over-bumping.

## Manifest pin syntax

Each consumer repo's `manifest.yaml` declares which versions it accepts:

```yaml
included:
  skills:
    peer-review: ^1.4.0          # caret: any 1.x.y where x.y >= 4.0
    audit-docs: ~1.2.3           # tilde: any 1.2.x where x >= 3
    build-task-bundle: 1.4.2     # exact pin
    capture-evidence: latest     # always latest (use sparingly)
```

| Operator | Range | Use when |
|---|---|---|
| `^MAJOR.MINOR.PATCH` | Any patch and minor within the same major | Default. Get bug fixes and additions automatically; avoid breaks. |
| `~MAJOR.MINOR.PATCH` | Any patch within the same minor | Maximum stability while still getting fixes. |
| `MAJOR.MINOR.PATCH` (exact) | Exactly this version | Locked production deployments. |
| `latest` | Whatever's newest in the registry | Active development; avoid in client repos. |

The Librarian's `librarian sync` command resolves pins against the registry's published versions and materializes the resolved versions into `base/`.

### Lockfile

A separate `.agents/manifest.lock.yaml` records the exact resolved versions at last sync. Like a package-manager lockfile, this guarantees reproducibility — two clones of the same repo at the same commit get the same `base/` content even if the registry has moved.

```yaml
# Generated by librarian sync; do not edit by hand.
budai-version: 0.4.2
resolved:
  skills:
    peer-review: 1.4.7         # ^1.4.0 resolved to 1.4.7
    audit-docs: 1.2.5          # ~1.2.3 resolved to 1.2.5
    build-task-bundle: 1.4.2   # exact
    capture-evidence: 1.0.4    # latest at sync time
```

The lockfile is committed. `librarian sync` updates it; consumers see updates as a normal commit and can review before merging.

## Stats versioning

Per `07-runtime-data.md`, every skill invocation is recorded with its version. `stats/skills.json` discriminates by `skill@version`:

```json
{
  "skills": {
    "peer-review@1.4.6": {"invocations": 60, "success-rate": 0.97},
    "peer-review@1.4.7": {"invocations": 12, "success-rate": 0.95}
  }
}
```

Why this matters:

- **A/B comparison.** When v1.4.7 ships, stats let you see whether the new version actually improved success rate before promoting to base.
- **Regression detection.** A patch bump that drops success rate by 10% is suspicious; the Librarian flags it.
- **Trend lines.** Over time, you can see whether a skill is getting better or accumulating cruft.

Stats persist forever (per the runtime-data retention policy). You can answer "what was peer-review's success rate during Q1 2026?" months later.

## Migration: handling breaking changes

When a skill goes major, consumer repos with `^OLD-MAJOR` pins don't auto-upgrade. They keep working on the old major version until someone explicitly bumps the manifest.

The migration flow:

1. **Registry releases v2.0.0** of the skill, with `breaking-changes-from: 1.5.4` set.
2. **Registry's CHANGELOG.md** documents the breaking changes and provides a migration guide.
3. **Consumer repos see notification.** Their next `librarian sync` shows: "peer-review v2.0.0 available; current pin `^1.4.0` doesn't upgrade. See migration guide."
4. **Repo maintainer reads migration guide.** Decides when to upgrade.
5. **Repo maintainer updates manifest.** Changes pin to `^2.0.0`. Resolves any local content that depended on the old contract (e.g., a local skill that called peer-review with the old input shape).
6. **`librarian sync`** materializes v2.0.0 into `base/`.
7. **Lockfile updates.** Commit the manifest + lockfile change as a single PR through the consumer repo's normal review process.

### The migration guide format

In the registry repo, breaking changes get a migration entry in `CHANGELOG.md`:

```markdown
## peer-review 2.0.0

**Breaking:** input `council-attempts-dir` renamed to `attempts-source` and now accepts either a directory path or a list of file paths (was: directory only).

### Migration
- If your manifest pins `peer-review: ^1`, update to `^2`.
- If your `local/skills/` has a wrapper around peer-review, change `council-attempts-dir` to `attempts-source` in the wrapper's input list.
- Local skill files invoking peer-review need their input mapping updated.
- No changes needed to base/ files; the registry handles those.

### Why
The new input shape supports compositions where the Judge wants to peer-review a synthesized subset of attempts (e.g., post-failure-loop survivors) without restructuring directories.
```

## Deprecation flow

A skill can be deprecated without immediate removal. Steps:

1. **Mark as deprecated.** Frontmatter changes `stability: deprecated`. Body adds a "Deprecation notice" section explaining why and what to use instead.
2. **Sync surfaces the warning.** Consumer repos doing `librarian sync` see a deprecation notice in the sync output for any deprecated skill in their pins.
3. **Stats track usage.** If usage continues, the deprecation is informational. If usage drops to zero across all repos for ≥6 months, the registry can move the skill to `archived/`.
4. **Removal.** Archived skills aren't returned by `librarian sync`. Manifests pinning archived skills get a hard error at sync time.

Deprecation never auto-removes. Removal is always explicit, always preceded by usage evidence.

## Multi-version coexistence

During a transition period, both old and new majors of a skill can coexist in the registry. The registry layout supports this:

```
base/skills/
├── peer-review.md           # current major (e.g., v2)
├── peer-review-v1.md        # legacy major during transition
└── archived/
    └── peer-review-v0.md    # old archived versions
```

The manifest can pin to a legacy variant for repos that haven't migrated:

```yaml
included:
  skills:
    peer-review: peer-review-v1@^1.5.0   # using the legacy file
```

Once the legacy file's usage drops to zero, it moves to `archived/`. This gives consumer repos a buffer window without forcing immediate upgrades.

## Rollback

If a new version turns out to be regressive after promotion to the registry:

1. **Hot-fix the bad version.** Issue a patch (e.g., 1.4.8) addressing the regression.
2. **Hard-rollback if hot-fix isn't feasible.** Mark the bad version as `stability: deprecated` AND remove from default sync. Consumer repos pinning `^1.4.0` resolve to the prior good version.
3. **Lockfiles can override.** A consumer repo with the bad version in its lockfile can manually downgrade by deleting the lock entry and re-syncing.

Rollbacks are rare. They're noted in CHANGELOG with a `## Rolled back` heading so the history is honest.

## What versioning is NOT

- Not a substitute for testing. A patch labeled compatible can still regress; stats catch what tests don't.
- Not a guarantee of forward compatibility. Old consumers using a new version of a skill they've cached might break if the runner has changed assumptions.
- Not a replacement for explicit deprecation cycles. Major bumps should be rare; deprecate first when possible.
- Not gated by registry maintainers being available. The registry is plain git; consumer repos with publish access can land changes via PR.
