# 09 — Bundle format

A **bundle** is a single self-contained markdown file the Implementer reads to know everything about a task. It's the data contract between the Librarian (which builds it) and the Planner / Implementer (which consume it).

Goal: 95% context coverage in one read. The Implementer can `Read` more files if needed, but most tasks shouldn't require it.

## Lifecycle

- **Created** when a task enters the workflow (Step 2 of the journey, see `08-the-journey.md`). The Librarian invokes the `build-task-bundle` skill.
- **Read** by the Planner first (during Step 3) and by every Implementer instance during fan-out (Step 6).
- **Re-built** if the task is edited materially before plan approval, or on explicit `bin/librarian bundle <task-id>`.
- **Archived** with the task — moves to `tasks/archive/` alongside the task definition.

## File location

`tasks/open/<NNN>-<slug>.bundle.md`, next to the task definition. The bundle's filename is the task's filename plus `.bundle.md`. One-to-one mapping; a task with no bundle hasn't entered Step 2 yet.

## Structure

A bundle has two parts: YAML frontmatter (the manifest) and a markdown body (the concatenated content). The frontmatter is the contract; the body is the payload.

```markdown
---
<YAML manifest>
---

# Bundle for Task 042: <task title>

## Source files
...

## Docs
...

## Decisions
...

## Related tasks
...
```

## YAML manifest schema

Locked schema. Validation runs at hand-off (Bundle → Planner) per `05-workflows.md`. An invalid manifest causes the Librarian to re-run.

```yaml
---
task-id: 042
generated: 2026-05-08T10:30:00Z
generator-version: 0.4.2          # version of the build-task-bundle skill
budget:
  target-tokens: 80000
  actual-tokens: 76420
  status: ok                      # ok | overflow | trimmed
included:
  source-files:
    - path: src/renderer/components/SpatialCanvas.tsx
      tokens: 4200
      reason: files-to-touch
    - path: src/main/index.ts
      tokens: 6800
      reason: ipc-host
  docs:
    - path: docs/architecture.md
      tokens: 3100
      reason: scope-match
    - path: src/renderer/components/README.md
      tokens: 800
      reason: dir-readme
  decisions:
    - path: memory/decisions/0003-webview-over-iframe.md
      tokens: 1200
      reason: scope-match
  related-tasks:
    - path: tasks/archive/038-webview-context-menu.md
      tokens: 1400
      reason: same-area-prior-work
  conventions:
    - section: ipc-naming
    - section: error-handling
  glossary:
    - widget
    - node
    - webview
referenced-but-not-included:      # only present when budget.status != ok
  - path: src/renderer/components/BrowserWidgetNode.tsx
    tokens: 5400
    reason: similar-pattern
    hint: "Read if you need a precedent for webview lifecycle in a node."
  - path: src/renderer/components/CanvasesWidgetShapeUtil.tsx
    tokens: 3200
    reason: legacy-pattern
    hint: "Probably not needed; only relevant if you touch the legacy ShapeUtil migration."
---
```

### Field reference

- **task-id** — the task's numeric ID. Must match the parent task's frontmatter.
- **generated** — ISO-8601 UTC timestamp when the bundle was built.
- **generator-version** — version of the `build-task-bundle` skill. Used by the Librarian to detect when a bundle was generated by a stale skill version and might need rebuilding.
- **budget** — token budget metadata.
  - `target-tokens` — the budget the bundler aimed for. Default 80000.
  - `actual-tokens` — the bundle's actual token count.
  - `status` — `ok` (under budget), `trimmed` (overflow handled gracefully), `overflow` (budget exceeded but bundler didn't trim — should never appear in shipped bundles; indicates a bundler bug).
- **included** — what's in the bundle, by category.
  - **source-files** — production code files. Each entry: path, tokens, reason for inclusion.
  - **docs** — markdown docs from `docs/` and READMEs from `src/`.
  - **decisions** — ADRs from `memory/decisions/`.
  - **related-tasks** — closed tasks in the same area, useful for precedent.
  - **conventions** — names of conventions sections pulled (not full path; the conventions file is split into named sections).
  - **glossary** — terms whose definitions are pulled. Only terms appearing in the task body.
- **referenced-but-not-included** — only present when budget overflowed. Each entry has the same fields as `included`, plus `hint:` — a one-line note from the Librarian explaining when to fetch it.

### Reason taxonomy

The `reason:` field is constrained to a small vocabulary so the Implementer can quickly skim why each file is in the bundle:

- `files-to-touch` — explicitly listed in the task's frontmatter.
- `direct-import` — imported by a `files-to-touch` file.
- `ipc-host` / `ipc-client` — touches the IPC channel the task is changing.
- `scope-match` — file's `@purpose` or directory matches the task's `scope:` field.
- `dir-readme` — README of a directory containing changed files.
- `same-area-prior-work` — a closed task in the same scope from the last 90 days.
- `similar-pattern` — heuristically similar to files-to-touch (used in overflow hints).
- `legacy-pattern` — older pattern the new work might need to interop with or migrate from.
- `keyword-match` — task body keyword appears in file's header.

New reasons can be added as the bundler matures, but each one should be portable across repos. A reason that's CanvasOS-specific shouldn't ship in the base skill.

## Body structure

The body is the concatenated content. Top-level sections in fixed order:

1. **Bundle header** — task summary, brief navigation, link back to task file.
2. **Source files** — each file as `### <path>` followed by a fenced code block of its contents, preceded by its header comment.
3. **Docs** — each doc as `### <path>` followed by full markdown.
4. **Decisions** — each ADR as `### <path>` followed by full markdown.
5. **Related tasks** — each task file as `### <path>` followed by frontmatter + body.
6. **Conventions** — each pulled section as `### <section-name>` followed by section content.
7. **Glossary** — each term as `### <term>` followed by its definition.
8. **Footer** — if `referenced-but-not-included` is present, repeat its content in human-readable form so the Implementer doesn't have to parse the YAML.

Section order is fixed because the Implementer reads top-to-bottom; relevance order matters. Source files come first because they're what the Implementer is editing; conventions and glossary come last because they're fallback reference.

## Token budget

### Why a budget

Sonnet's context window is 200k tokens. A bundle that consumes 150k leaves 50k for the working transcript — diff being written, intermediate reasoning, tool outputs, the task description being re-read. Cramped windows hurt attempt quality.

Default: target 80k tokens for the bundle. Leaves 120k for the working transcript. Configurable per-task via `bundle-budget: <N>` in task frontmatter.

### Estimation

The bundler estimates tokens via chars/4 rule of thumb. Imprecise but cheap. The actual count is computed by the runner once the bundle is materialized; if it exceeds target, the bundler is invoked again with a tighter target until convergence.

### Priority order

The bundler adds files in priority order until the budget is reached:

1. `files-to-touch` (explicit)
2. `direct-import` (one-hop dep graph from files-to-touch)
3. `ipc-host` / `ipc-client` (when task touches IPC)
4. `scope-match` ADRs from `memory/decisions/`
5. Directory READMEs touching changed files
6. Conventions sections relevant to scope keywords in task
7. Glossary terms appearing in task body
8. `same-area-prior-work` related tasks
9. Speculative additions (`similar-pattern`)

When the bundler can't fit a category-N file, it stops adding that category and moves to the next, looking for smaller files. When even that fails, remaining candidates move to `referenced-but-not-included`.

### Overflow

When budget is reached:

1. Stop adding to `included`.
2. For each remaining candidate that wasn't added, write an entry in `referenced-but-not-included` with a one-line `hint:` from the Librarian explaining when to fetch.
3. Set `budget.status: trimmed`.
4. The body's footer renders the overflow list in human-readable form.

The Implementer is told (via its role definition) to consult `referenced-but-not-included` if stuck. Most tasks shouldn't need to.

## Relevance scoring

The bundler scores files using:

- **Direct match** — file appears in `task.files-to-touch`. Score: 100.
- **Import proximity** — file is imported by a `files-to-touch` file (score 80) or transitively at distance 2 (score 50). Beyond distance 2, the score drops below the inclusion threshold by default.
- **Scope match** — file's `@purpose` or `@role` header matches `task.scope`. Score: 60.
- **Keyword match** — task title and body words appear in file's `@purpose` or filename. Score: 40 per keyword, capped at 80.
- **Header gotchas** — files with `@gotchas` get +10 because non-obvious behavior matters when changing nearby code.
- **Stability penalty** — `@stability: deprecated` gets -30 (don't pull deprecated patterns into bundles unless explicitly relevant).

Scores are ordinal, not absolute. The bundler picks the top N by score until budget is exhausted.

## Validation

A bundle is valid if:

1. YAML frontmatter parses.
2. `task-id` matches the task file's frontmatter.
3. All paths in `included` resolve to existing files.
4. `actual-tokens` <= 1.10 × `target-tokens` (10% slack for estimation imprecision).
5. `included.source-files` has at least one entry, OR the task is type `audit` / `research` (which legitimately have no source-files-to-touch).
6. Body sections appear in the spec'd order.

Validation runs in the runner at hand-off. A failed bundle is rebuilt with the failure reason as input to the bundler.

## What a bundle is NOT

- Not a freeze of repo state. The bundle is a snapshot at generation time; if main moves, the bundle is stale. Stale bundles trigger rebuild.
- Not a substitute for `Read`. The Implementer can still read other files; the bundle is the *expected* context, not a hard limit.
- Not the task itself. The task lives in `tasks/open/<id>.md`; the bundle is a separate file. Implementers read both.
- Not personalized per Implementer instance. All Implementer instances in a fan-out get the same bundle.
