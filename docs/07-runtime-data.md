# 07 — Runtime data

**Runtime data** is everything produced by agents during execution: transcripts, diffs, evidence, council artifacts, message-board posts, stats. Lives in `.agents/{runs,council,messages,stats,index}/` locally and streams to the ultimate-widget backend for cross-repo aggregation and queryability.

This document specifies the on-disk shape, the streaming model, the backend mapping, and the retention policies.

## Top-level layout

```
.agents/
├── runs/                # per-agent-invocation artifacts
├── council/             # per-task multi-attempt records
├── messages/            # async coordination posts
├── stats/               # auto-generated aggregations
└── index/               # auto-generated navigation aids
```

All five directories are gitignored locally. They stream to the backend continuously. They're regenerable from backend records if a local repo is wiped.

## `runs/<run-id>/`

One directory per agent invocation. The `<run-id>` is a ULID/UUIDv7 generated at dispatch.

```
runs/<run-id>/
├── meta.json            # who/what/when of this run
├── transcript.md        # the full agent transcript
├── diff.patch           # diff produced by this run (if any)
├── failure.md           # only present if this run failed (per 14-failure-loop.md)
├── worktree/            # the git worktree for Implementer runs (cleaned up post-task)
└── evidence/            # only for Verifier runs (per 13-evidence-capture.md)
```

### `meta.json` schema

```json
{
  "run-id": "01HX2Y3Z-7f2c-...",
  "task-id": 42,
  "opaque-id": "attempt-A",
  "role": "implementer",
  "skill-invocations": [
    {"skill": "run-preflight", "version": "1.0.2", "started": "...", "ended": "...", "tier": "haiku"}
  ],
  "model": "claude-sonnet-4-6",
  "runner": "claude-code",
  "tier": "sonnet",
  "started": "2026-05-08T11:00:01Z",
  "ended": "2026-05-08T11:18:32Z",
  "status": "completed",       // pending | completed | failed | aborted
  "exit-code": 0,
  "tokens": {
    "input": 76420,
    "output": 4180,
    "cache-read": 71200
  }
}
```

Written incrementally during the run; finalized at completion.

### `transcript.md`

The full agent transcript. Format depends on the runner — Claude Code's transcript shape differs from Codex's. Each runner shim normalizes to a common structure for the streaming layer (per `15-framework-agnostic.md`).

## `council/<task-id>/`

Per-task multi-attempt records. Full schema in `12-isolation-and-fanout.md`. Summary:

```
council/<task-id>/
├── dispatch.json
├── attempts/
│   ├── attempt-A.md
│   └── attempt-A.patch
├── reviews/
├── verifier-reports/
├── mapping.json
└── verdict.md
```

Runtime-data note: the council directory is the durable side of fan-out. It's the audit anchor — given any task ID, the entire decision trail is reconstructible from this directory plus the linked `runs/<run-id>/` records.

## `messages/`

Async coordination — agents posting to channels and threading replies. Inspired by agenthub's message-board pattern (`01-design-principles.md`, principle 1's lineage).

```
messages/
├── channels/
│   ├── tasks.md         # task assignments and status updates
│   ├── review.md        # review requests and verdicts
│   ├── ops.md           # Librarian sweep deltas
│   ├── decisions.md     # ADR-related coordination
│   └── escalations.md   # cross-role escalations (per 14-failure-loop.md)
└── threads/
    └── <thread-id>.md
```

### Channel post format

Each channel file is append-only:

```markdown
## 2026-05-08T11:00:00Z | router | task-dispatch
Dispatched fan-out 3 on task 42 (ship-feature workflow).
Opaque IDs: attempt-A, attempt-B, attempt-C.
See council/42/dispatch.json.

---

## 2026-05-08T11:18:32Z | implementer-instance-7f2c | submission
attempt-A submitted on task 42. See council/42/attempts/attempt-A.md.

---

## 2026-05-08T11:35:00Z | verifier | escalation-notice
Escalated AC2 ambiguity on task 42 to Planner.
See messages/threads/escalation-42-ac2.md.

---
```

Format: H2 header with `<ISO timestamp> | <author> | <message-type>`, body, `---` separator.

### Threads

When a coordination conversation goes deeper than a single post, it spawns a thread file:

```markdown
# Thread: escalation-42-ac2

## 2026-05-08T11:35:00Z | verifier
AC2 says "commands execute" but doesn't define "command" — shell, CLI tool, or both?

## 2026-05-08T11:36:42Z | planner
Both. Updating plan with explicit AC2.1 and AC2.2.

## 2026-05-08T11:38:11Z | router
Plan revised; relaunching attempts. New opaque IDs in council/42/dispatch-2.json.
```

Threads are referenced from the main channel via `See messages/threads/<id>.md`. Threads themselves are append-only.

## Streaming to the backend

The ultimate-widget backend is the durable, queryable, cross-repo store. Local `.agents/` directories are caches plus working memory; the backend is the source of truth for analytics.

### What gets streamed

| Source | Backend table | Frequency |
|---|---|---|
| `runs/<run-id>/meta.json` | `runtime_sessions` | On run completion |
| `runs/<run-id>/transcript.md` | (referenced; stored externally if large) | On run completion |
| `runs/<run-id>/evidence/ac-mapping.json` | `runtime_evidence` | On Verifier completion |
| `runs/<run-id>/evidence/screenshots/`, etc. | (object storage; references in `runtime_evidence`) | On Verifier completion |
| `council/<task-id>/dispatch.json` | `runtime_councils` | On dispatch |
| `council/<task-id>/attempts/*.md` | `runtime_attempts` | On attempt submission |
| `council/<task-id>/reviews/*.md` | `runtime_reviews` | On review submission |
| `council/<task-id>/verdict.md` | `runtime_verdicts` | On verdict |
| `messages/channels/*.md` (deltas only) | `runtime_messages` | Continuous |
| `stats/*.json` | (recomputed server-side from above) | N/A; backend regenerates |

Skill invocations get their own row per run:

| Source | Backend table |
|---|---|
| `runs/<run-id>/meta.json:skill-invocations` (each item) | `runtime_skill_invocations` |

This is what makes per-skill, per-version stats queryable across repos.

### Streaming mechanism

`bin/librarian sync-out` pushes deltas to the backend. Two modes:

- **Push on completion** — happens on every role completion, semi-real-time.
- **Batch** — daily fallback if push fails or if the backend was unreachable.

Idempotency: every record has a stable key derived from `(repo, run-id)` or `(repo, task-id)`. Re-pushing the same record is a no-op.

### What does NOT get streamed

- Per-task working transcripts when they exceed a size threshold (default 100k tokens) — a reference is streamed; the body stays local. Configurable.
- Personal-layer memory (`~/.claude/memory/`) — never streamed; that's outside budai's scope per `06-memory.md`.
- Worktree contents — they're transient by design; only the diff matters and that's already streamed.
- Untouchables and ADRs — those are in git, not runtime data; the backend has read access via the registry's git mirror but doesn't ingest them as runtime records.

## Backend table schema (sketch)

Aligning with the existing ultimate-widget runtime sync engine (per the original MVP plan), extended for budai:

- `runtime_sessions` — one row per agent invocation (`run_id`, `task_id`, `role`, `model`, `runner`, `tier`, `started`, `ended`, `status`, `tokens_*`).
- `runtime_run_events` — events emitted during a run (`run_id`, `sequence_no`, `event_type`, `payload`). Event types: `attempt.submitted`, `review.posted`, `verdict.rendered`, `lesson.promoted`, `skill.invoked`.
- `runtime_councils` — one row per fan-out (`task_id`, `dispatched_at`, `fan_out`, `workflow`, `winning_attempt`).
- `runtime_attempts` — one row per attempt within a council (`task_id`, `opaque_id`, `run_id`, `status`, `submitted_at`).
- `runtime_reviews` — one row per peer review.
- `runtime_verdicts` — one row per task's final verdict.
- `runtime_evidence` — one row per AC's evidence pointer (`run_id`, `ac_id`, `evidence_path`, `claim`, `verdict`).
- `runtime_skill_invocations` — one row per skill invocation (`run_id`, `skill`, `version`, `tier`, `started`, `ended`, `success`).
- `runtime_messages` — one row per channel post / thread reply (`channel_or_thread`, `posted_at`, `author`, `message_type`, `body`).

Schema migrations to introduce these tables are out of scope for this doc; see the ultimate-widget backend repo for migration mechanics.

## Local retention

Defaults, all configurable:

| Path | Retention | Trigger for cleanup |
|---|---|---|
| `runs/<run-id>/transcript.md` | 30 days | Monthly Librarian sweep |
| `runs/<run-id>/diff.patch` | 90 days | Monthly Librarian sweep |
| `runs/<run-id>/evidence/` | 30 days for losers; permanent for winners | Monthly Librarian sweep |
| `runs/<run-id>/worktree/` | Until task closes | Judge integration step |
| `council/<task-id>/` | Permanent until task is archived for 365 days | Annual Librarian sweep |
| `messages/channels/*.md` | Truncated to last 30 days; older content rotated | Weekly Librarian sweep |
| `stats/*.json` | Permanent (regenerated; never deleted) | Never |
| `index/*` | Permanent (regenerated) | Never |

Cleanup respects backend confirmation: a record is cleaned locally only if the backend confirmed it received that record's stream.

## Privacy / sensitive data

The streaming layer redacts known patterns:

- API tokens (`sk-`, `gh_`, `github_pat_`, etc.) → `<REDACTED-TOKEN>`
- Email addresses in transcripts (configurable per repo) → `<REDACTED-EMAIL>`
- File paths under `~/.ssh/`, `~/.aws/`, etc. → `<REDACTED-CREDENTIALS-PATH>`

Repos with stricter policies can extend the redaction list in `local/conventions.md` under "Sensitive data."

For client work, the stricter posture is: the backend instance is private to the org (not Dopomogai's central one), evidence streams are encrypted at rest, and retention is tightened.

## Re-derivation

If a local `.agents/{runs,council,messages,stats}/` is lost (corruption, accidental delete, fresh clone), the Librarian can re-pull from the backend:

```bash
bin/librarian sync-in --since=last-month
```

Pulls the records the backend has for this repo. Reconstructs local files from streamed records. Stats are regenerated from primary records.

The git-tracked artifacts (`memory/`, `local/`, `manifest.yaml`, source headers, READMEs) come back via `git pull`. Nothing is lost permanently — provided either git or the backend is intact.

## What runtime data is NOT

- Not config. Manifest and role/skill/workflow files are config; they live in git.
- Not memory. Memory is the durable knowledge layer (`06-memory.md`); runtime data is the trace layer that *feeds* memory via the Librarian's promotion logic.
- Not the source of truth for "what the system does." That's the role/skill/workflow files. Runtime data is the source of truth for "what the system did."
- Not user data in the analytics sense. We don't track end-user behavior; we track agent execution.
